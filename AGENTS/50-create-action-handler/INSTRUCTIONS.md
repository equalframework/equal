1. Identify the correct mechanism before writing anything: `actions/` for an autonomous/mutating application endpoint, `getActions()` for an action carried by a selected entity, and `getWorkflow()` + `core_model_transition` if it is only a status transition.
2. Do not create a file in `actions/` if the need is an action on a collection of objects already selected in a view; in that case, follow the `getActions()` + `doXxx()` method + `core_model_do` button pattern.
3. Inspect similar examples in the same package or on a neighboring entity before choosing the pattern; in this repository, `documents\Document::start_processing`, `documents\processing\DocumentProcess::perform_*`, `purchase\supplier\Suppliership::sync_bank_accounts`, and `hr/actions/employee/Employee/create-user.php` provide the actual conventions.
4. Locate the target business class and check whether it already inherits parent actions; if so, extend using `array_merge(parent::getActions(), [...])` rather than rewriting the entire map.
5. Name the action with an explicit and stable verb, using `snake_case` in `getActions()` and `doPascalCase` for the method; repository names follow patterns like `start_processing`, `perform_matching`, `sync_from_teams`.
6. Declare the inline action in `getActions()` with at least `description`, `policies`, and `function`; `help` may be added if the action is not obvious.
7. Clearly define the input contract: for `getActions()`, the main input is the current collection (`$self`) and possibly dependencies injected by the framework based on parameter names (`$auth`, `$orm`, `$context`, `$dispatch`, `$lang`, `$values`); for an `action handler`, parameters must be declared in `eQual::announce()`.
8. If the action must accept one or multiple objects via HTTP, follow the repository handler pattern with `id` and optionally `ids`, then normalize the identifier list before processing.
9. Move business preconditions into policies whenever possible; in this repository, inline actions are often gated by `can_*` or `is_*`, and these policies return a structured map per object and per error key.
10. Return stable and explicit business error keys, not just free-form messages; examples use keys such as `already_processing`, `missing_due_date`, `invalid_invoice_status`.
11. Reserve `trigger_error("APP::...")` for logging, warnings, and diagnostics; for expected functional blocking, prefer a policy or a domain exception consistent with the flow conventions.
12. Explicitly load the required fields before processing the action using `$self->read([...])`, then iterate over the collection; repository actions are designed to work on one or multiple ids, not just a singleton.
13. Do not duplicate logic already present in another action, transition, `onupdate`, or business method; if the behavior already exists, chain with `->do('...')` or call the appropriate existing entry point.
14. If the inline action is triggered from a view, the view must use `controller: "core_model_do"` with `params.entity`, `params.action`, and generally `ids: []`; if it is a workflow transition, use `core_model_transition`, not `core_model_do`.
15. Add `visible` only when the action must be exposed in the UI, and `confirm: true` only for genuinely sensitive actions; the repository does not systematically add a view button for every existing action.
16. For an `action handler`, follow the full `eQual::announce()` structure: `description`, `params`, `access` if needed, `response`, `providers`, `constants`, then end with `httpResponse()->status(...)->send()`; handlers in the repository often return `201` or `204`.
17. For an inline action invoked via `core_model_do`, do not build a specific HTTP payload unless necessary; in this repository, such actions mainly mutate business state and rely on standard refresh behavior.
18. Update translations when the action is exposed to the user: at minimum `view -> form.* -> actions -> action.xxx -> label/description`, and include business errors in the `error` block when they must be translatable in the UI.
19. Check business side effects on related objects; in this repository, several actions synchronize or create related objects, so these dependencies must be explicit and consistent.
